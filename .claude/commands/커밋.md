# 커밋 파이프라인

다음 단계를 순서대로 실행하라.

## 1단계: 변경사항 분석

아래 명령어를 병렬로 실행해 현재 상태를 파악하라.
- `git status`
- `git diff HEAD` (스테이지 포함 전체 diff)
- `git log --oneline -5` (최근 커밋 스타일 파악)
- `ls docs/adr/ 2>/dev/null || echo "no adr yet"` (기존 ADR 번호 파악)

## 2단계: 변경 유형 분류 및 커밋 분리 판단

분석한 내용을 아래 기준으로 분류하라.

- **환경/설정 변경**: package.json, metro.config.js, app.json, eas.json, .env, 빌드 설정 등
- **버그 수정**: 오류 해결, 동작 이상 수정
- **기능 추가/변경**: 새 화면, 컴포넌트, 로직 추가
- **리팩토링**: 코드 구조 개선 (동작 변화 없음)
- **문서**: README, 주석, docs/ 파일 등
- **도구/설정**: .claude/, 스크립트, CI 설정 등

### 커밋 분리 원칙

변경사항이 **서로 다른 관심사**에 속하면 반드시 **별도 커밋으로 분리**하라.

**분리해야 하는 경우:**
- 버그 픽스 + 새 기능이 함께 있을 때
- 소스 코드 변경 + 도구/파이프라인 설정 변경이 함께 있을 때
- 서로 독립적인 두 가지 버그 픽스가 함께 있을 때

**분리 방법:**
관심사별로 파일을 묶어 `git add [파일들]` 후 커밋, 이를 반복하라.
각 커밋마다 해당 변경에 맞는 문서(troubleshooting/CHANGELOG/ADR)를 함께 포함시켜라.

분리가 필요한 경우, 커밋 전에 사용자에게 분리 계획을 먼저 보여주고 확인받아라.

## 3단계: ADR 필요 여부 판단

아래 기준으로 ADR이 필요한지 판단하라.

**ADR이 필요한 경우 (하나라도 해당하면):**
- 새로운 라이브러리/프레임워크를 도입했다
- 기술 스택 또는 아키텍처 방향이 바뀌었다
- 여러 대안 중 하나를 선택했다 (선택의 트레이드오프가 존재)
- 프로젝트 전반에 영향을 주는 설정/패턴을 새로 정했다
- 나중에 "왜 이렇게 했지?"라는 질문이 나올 만한 결정이다

**ADR이 필요 없는 경우:**
- 단순 버그 수정
- 기존 패턴을 따르는 기능 추가
- 의존성 버전 업데이트 (단순 호환성 픽스)
- 스타일/포맷 변경

### ADR이 필요하다고 판단한 경우

AskUserQuestion 도구를 사용해 사용자에게 다음을 질문하라.
한 번에 모든 질문을 묶어서 물어보라.

```
ADR을 작성할게요. 몇 가지 확인할게요:

1. 이 결정의 핵심 배경/이유가 무엇인가요? (왜 기존 방식을 바꿨나요?)
2. 고려했던 다른 대안이 있었나요? (없으면 "없음")
3. 이 결정의 단점이나 트레이드오프가 있나요? (없으면 "없음")
4. ADR 상태는 무엇인가요? (Accepted / Proposed / Experimental 중 선택)
```

사용자 답변을 받은 후 `docs/adr/` 폴더에 아래 형식으로 파일을 생성하라.
파일명: `NNN-제목-요약.md` (NNN은 기존 ADR 개수 + 1, 3자리 숫자)

```markdown
# ADR-NNN: 제목

- **날짜**: YYYY-MM-DD
- **상태**: [Accepted / Proposed / Experimental]

## 배경

왜 이 결정이 필요했는지. 어떤 문제를 해결하려 했는지.

## 결정

무엇을 어떻게 하기로 했는지. 핵심 내용.

## 고려한 대안

| 대안 | 장점 | 단점 |
|------|------|------|
| 현재 선택 | ... | ... |
| 대안 A | ... | ... |

## 결과

### 긍정적 효과
- ...

### 트레이드오프 / 단점
- ...

## 관련 파일
- 변경된 파일 목록
```

## 4단계: 문서 자동 업데이트

### 환경/설정 변경 또는 버그 수정이 포함된 경우
`docs/troubleshooting.md` 파일에 아래 형식으로 항목을 추가하라.
파일이 없으면 새로 생성하라.

```markdown
## [YYYY-MM-DD] 제목 (한 줄 요약)

### 문제
어떤 에러 또는 증상이 있었는지 설명.

### 원인
왜 발생했는지 설명.

### 해결
어떻게 고쳤는지 설명. 핵심 코드나 명령어 포함.

### 관련 파일
- 변경된 파일 목록
```

### 기능 추가/변경이 포함된 경우
`CHANGELOG.md` 파일에 아래 형식으로 항목을 추가하라.
파일이 없으면 새로 생성하라.

```markdown
## [Unreleased]

### Added
- 새로 추가된 기능

### Changed
- 변경된 기능

### Fixed
- 수정된 버그
```

두 유형이 모두 해당하면 두 파일 모두 업데이트하라.

## 5단계: 커밋 생성

1. 모든 문서 파일(troubleshooting.md, CHANGELOG.md, docs/adr/*.md)을 스테이징에 추가하라.
2. 변경된 소스 파일도 스테이징하라 (민감 정보 파일 제외).
3. 커밋 메시지를 작성하라:
   - 첫 줄: 변경의 **목적** 중심 (영어 또는 한국어)
   - ADR이 생성된 경우 본문에 `docs: add ADR-NNN` 언급
   - 필요시 본문에 세부 내용 추가
4. 커밋을 실행하라.

$ARGUMENTS 가 있으면 커밋 메시지 힌트로 활용하라.
