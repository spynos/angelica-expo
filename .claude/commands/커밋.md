# 커밋 파이프라인

[Conventional Commits](https://www.conventionalcommits.org/) + [Keep a Changelog](https://keepachangelog.com/) 표준을 따른다.
다음 단계를 순서대로 실행하라.

## 1단계: 변경사항 분석

아래 명령어를 병렬로 실행해 현재 상태를 파악하라.
- `git status`
- `git diff HEAD` (스테이지 포함 전체 diff)
- `git log --oneline -5` (최근 커밋 스타일 파악)
- `ls docs/adr/ 2>/dev/null || echo "no adr yet"` (기존 ADR 번호 파악)

## 2단계: 변경 유형 분류 (Conventional Commits type)

각 변경 묶음을 아래 type 중 하나로 분류하라.

| Type | 설명 | CHANGELOG 기록 |
|------|------|----------------|
| `feat` | 새 기능 추가 | **Added** |
| `fix` | 버그 수정 | **Fixed** |
| `perf` | 성능 개선 | **Changed** |
| `refactor` | 동작 변화 없는 리팩토링 | 생략 |
| `docs` | 문서만 변경 | 생략 |
| `test` | 테스트만 추가/수정 | 생략 |
| `build` | 빌드 시스템/의존성 변경 | **Changed** (사용자·빌드 영향 시) |
| `ci` | CI 설정 변경 | 생략 |
| `chore` | 기타 잡일 | 생략 |
| `revert` | 이전 커밋 되돌림 | 원 커밋에 따라 |
| `style` | 포맷팅만 | 생략 |

**Breaking change**: type 뒤에 `!` 붙이거나 footer에 `BREAKING CHANGE: …` 기재.
CHANGELOG에서는 별도 강조 또는 `Removed` / `Deprecated` 섹션에 기록.

### 커밋 분리 원칙 (atomic commits)

변경사항이 **서로 다른 관심사**에 속하면 반드시 **별도 커밋으로 분리**하라.
각 커밋은 단일 논리 단위여야 한다.

**분리해야 하는 경우:**
- `fix` + `feat` 혼재
- 소스 코드 변경 + 도구/파이프라인 설정 변경 혼재
- 서로 독립적인 두 버그 수정

**분리 방법:**
관심사별로 `git add [파일들]` 후 커밋, 반복. 각 커밋에 해당 변경의 문서(CHANGELOG/troubleshooting/ADR)도 같이 포함.

분리가 필요한 경우 커밋 전에 사용자에게 분리 계획을 보여주고 확인받아라.

## 3단계: ADR 필요 여부 판단

**ADR이 필요한 경우 (하나라도 해당):**
- 새로운 라이브러리/프레임워크 도입
- 기술 스택 또는 아키텍처 방향 변경
- 여러 대안 중 하나 선택 (트레이드오프 존재)
- 프로젝트 전반에 영향을 주는 설정/패턴 확립
- 나중에 "왜 이렇게 했지?" 질문이 나올 만한 결정

**ADR이 필요 없는 경우:**
- 단순 버그 수정
- 기존 패턴을 따르는 기능 추가
- 의존성 버전 업데이트 (단순 호환성 픽스)
- 스타일/포맷 변경

### ADR이 필요한 경우

AskUserQuestion 도구로 한 번에 묶어 질문하라:

```
ADR을 작성할게요. 몇 가지 확인할게요:

1. 이 결정의 핵심 배경/이유가 무엇인가요?
2. 고려했던 다른 대안이 있었나요? (없으면 "없음")
3. 이 결정의 단점이나 트레이드오프가 있나요? (없으면 "없음")
4. ADR 상태는 무엇인가요? (Accepted / Proposed / Experimental)
```

답변 후 `docs/adr/NNN-제목-요약.md` 생성 (NNN은 기존 ADR 개수 + 1, 3자리):

```markdown
# ADR-NNN: 제목

- **날짜**: YYYY-MM-DD
- **상태**: [Accepted / Proposed / Experimental]

## 배경

왜 이 결정이 필요했는지. 어떤 문제를 해결하려 했는지.

## 결정

무엇을 어떻게 하기로 했는지. 핵심 내용.

## 고려한 대안

| 대안 | 장점 | 단점 |
|------|------|------|
| 현재 선택 | ... | ... |
| 대안 A | ... | ... |

## 결과

### 긍정적 효과
- ...

### 트레이드오프 / 단점
- ...

## 관련 파일
- 변경된 파일 목록
```

## 4단계: 문서 자동 업데이트

### CHANGELOG.md (Keep a Changelog 1.1.0 표준)

**기록 대상 — 사용자/빌드/배포에 영향 있는 변경만:**

| 섹션 | 언제 |
|------|------|
| **Added** | 새 기능, 새 화면, 새 API, 새 설정 |
| **Changed** | 기존 동작 변경, 의존성 메이저/마이너 업그레이드, 성능 개선 |
| **Deprecated** | 곧 제거 예정으로 표시된 기능 |
| **Removed** | 삭제된 기능/API |
| **Fixed** | 버그 수정 |
| **Security** | 취약점 수정 |

**기록 제외 — 사용자 무관한 내부 변화:**
- `docs`, `test`, `style`, `refactor`, `chore`, `ci` type 커밋
- 단순 lockfile 재생성
- 동작 변화 없는 리팩토링

**작성 언어 규칙:**
- 섹션 헤더는 영문 유지 (`Added` / `Changed` / `Deprecated` / `Removed` / `Fixed` / `Security`) — Keep a Changelog 파서 및 릴리즈 도구 호환성 위해.
- **각 항목 본문은 국문으로 작성.** 패키지명·파일명·버전 번호 같은 식별자는 원문 그대로.

CHANGELOG.md가 없으면 아래 형식으로 새로 생성:

```markdown
# 변경 로그 (Changelog)

이 프로젝트의 모든 주요 변경사항은 이 파일에 기록됩니다.

형식은 [Keep a Changelog](https://keepachangelog.com/ko/1.1.0/) 1.1.0을 따르며,
버전 관리는 [유의적 버전(SemVer)](https://semver.org/lang/ko/) 체계를 따릅니다.

## [Unreleased]

### Added
- ...

### Changed
- ...

### Fixed
- ...
```

기존 파일이 있으면 `[Unreleased]` 하위의 해당 섹션에 항목 추가. 비어 있는 섹션은 두지 말고 제거. 릴리즈 시 `[Unreleased]` → `[x.y.z] - YYYY-MM-DD`로 승격 (날짜는 ISO 8601).

### docs/troubleshooting.md (재발 가능한 이슈만)

**기록 대상:**
- 다른 개발자나 미래의 본인이 동일 문제를 다시 마주칠 가능성이 있는 이슈
- 에러 메시지 → 원인 → 해결 경로가 자명하지 않은 경우
- 외부 라이브러리/툴체인 resolver 버그 등 재현성 있는 환경 이슈

**기록 제외:**
- 일회성 업그레이드·마이그레이션 과정의 흔적 → CHANGELOG 또는 ADR로
- 스스로 만든 실수를 즉시 고친 경우
- 커밋 메시지와 코드만 봐도 맥락이 파악되는 변경

형식:

```markdown
## [YYYY-MM-DD] 제목 (한 줄 요약)

### 문제
에러 또는 증상.

### 원인
왜 발생했는지.

### 해결
어떻게 고쳤는지. 핵심 코드나 명령어 포함.

### 관련 파일
- 변경된 파일 목록
```

## 5단계: 커밋 생성 (Conventional Commits 형식)

### 커밋 메시지 형식

```
<type>(<scope>): <subject>

[optional body]

[optional footer]
```

- **type**: 2단계의 분류 (feat, fix, docs, build, chore, …)
- **scope**: 영향 범위 (선택). 예: `auth`, `router`, `deps`, `game`
- **subject**: 명령형 현재시제, 소문자 시작, 마침표 없음, 50자 이내 권장
- **body**: 변경의 **왜**를 설명. 줄 간격 한 줄, 72자 래핑 권장
- **footer**:
  - `BREAKING CHANGE: <설명>`
  - `Refs: #123` 또는 `Closes: #123`
  - `Co-Authored-By: …`

예시:

```
feat(auth): add Apple Sign-In callback handler
fix(router): resolve deep-link param encoding issue
build(deps): align dependencies to Expo SDK 55
refactor(game): extract shared storage layer
docs: add ADR-003 for SDK 55 upgrade decision
chore: regenerate package-lock.json
```

Breaking change 예시:

```
feat(api)!: replace session tokens with JWT

BREAKING CHANGE: clients must re-authenticate after upgrade.
```

### 실행 순서

1. 관련 문서(CHANGELOG.md, troubleshooting.md, docs/adr/*.md)를 스테이징에 추가.
2. 변경된 소스 파일도 스테이징 (민감 정보 파일 제외).
3. 위 형식에 맞춰 커밋 메시지 작성.
4. ADR 생성 시 body에 `Refs: docs/adr/NNN-*.md` 명시.
5. 커밋 실행.

$ARGUMENTS가 있으면 커밋 메시지 힌트로 활용하라.
